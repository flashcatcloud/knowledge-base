Flashduty periodically connects to an external database, executes SQL queries, and converts the results into standard alert events. Use this when alert data is stored in a database and cannot be pushed actively.

:::tips
DB Pull is the right choice when alert data is stored in a relational database, the monitoring system cannot send webhooks, or when you need to extract abnormal records directly from business tables as alerts. If your system already supports webhook delivery, prefer the corresponding push-based integration. It has lower latency and a smaller resource footprint.
:::

## In Flashduty
---

You can create a DB Pull integration through either of these two methods:

### Using Private Integration

Choose this method when you don't need to route alert events to different channels. It is simpler.

<details>
  <summary>Expand</summary>

  1. Go to the Flashduty console, select **Channel**, and enter a specific channel's details page.
  2. Select the **Integrations** tab, click **Add Integration** to enter the add integration page.
  3. Select the **DB Pull** integration.
  4. Fill in the form as described in "Configuration" below, then click **Save**.

</details>

### Using Shared Integration

Choose this method when you need to route alerts to different channels based on the alert event payload.

<details>
  <summary>Expand</summary>

  1. Go to the Flashduty console, select **Integration Center => Alert Events**, and enter the integration selection page.
  2. Select the **DB Pull** integration and fill in **Integration Name**.
  3. Configure **Default Route** and select the corresponding channel. After the integration is created, you can go to **Route** to configure more routing rules.
  4. Fill in the form as described in "Configuration" below, then click **Save**.

</details>
</div>

## Overview
---

<div class="md-block">

By configuring a database connection and SQL query, Flashduty periodically connects to the database on the schedule you define, executes the query, converts each result row into an event that conforms to the [standard alert event](/en/on-call/integration/alert-integration/alert-sources/standard%20alert) protocol through field mapping, and writes the event into the alert channel. Your system does not need to support push delivery, as long as the alert data is stored in a database that Flashduty can access.

### Use Cases

- Alert data is stored in MySQL, PostgreSQL, or ClickHouse
- The monitoring system does not support webhook push and alert data can only be retrieved by querying a database
- Abnormal records need to be extracted directly from business tables as alerts
- A custom system cannot be retrofitted for push mode yet, and you want to integrate quickly via pull

### Configuration Steps

1. Go to the Flashduty console, navigate to **Integration Center → Alert Events**, search for and select the **DB Pull** integration.
2. Configure database connection settings (see [Database Connection](#database-connection)).
3. Configure the SQL query used to pull alert data (see [SQL Query](#sql-query)).
4. Configure cursor columns to make sure each poll retrieves only incremental data (see [Cursor Configuration](#cursor-configuration)).
5. Configure field mapping and severity mapping (see [Field Mapping](#field-mapping) and [Severity Mapping](#severity-mapping)).
6. Click **Save** to enable the integration. You can also manually trigger a pull from the integration details page.

</div>

## I. Database Connection <span id="database-connection"></span>
---

<div class="md-block">

| Parameter | Required | Default | Description |
| :-: | :-: | :-: | :--- |
| **Database type** | Yes | - | Supports `MySQL`, `PostgreSQL`, and `ClickHouse`. |
| **Host** | Yes | - | Database server address, either an IP address or domain name. |
| **Port** | Yes | - | Database port, range `1 ~ 65535`. |
| **Username** | Yes | - | Database login username. |
| **Password** | No | - | Database login password. The system stores it encrypted. |
| **Database name** | Yes | - | Name of the database to connect to. |
| **Connection parameters** | No | - | Additional connection parameters in key-value format. For example, MySQL can use `charset=utf8mb4`, and PostgreSQL can use `sslmode=require`. |

:::warning
For security reasons, Flashduty non-private deployments cannot connect to private network addresses such as `127.0.0.1`, `10.x.x.x`, or `192.168.x.x`. Make sure your database is accessible from the public internet, or use a private deployment. We recommend creating a read-only account for Flashduty and granting only the SELECT permissions required for the target tables.
:::

</div>

## II. SQL Query <span id="sql-query"></span>
---

<div class="md-block">

| Parameter | Required | Default | Description |
| :-: | :-: | :-: | :--- |
| **Query** | Yes | - | SQL `SELECT` statement used to pull alert data. It must start with `SELECT` or `WITH`. |
| **Timeout (seconds)** | No | `5` | Timeout for a single query, range `1 ~ 10`. |
| **Polling cycle (seconds)** | Yes | `300` | Interval between successive polls. Minimum 30 seconds. |
| **Rows per page** | No | `500` | Maximum number of rows returned by each paginated query, range `1 ~ 1000`. |
| **Max pages** | No | `5` | Maximum number of paginated queries to run in a single polling cycle, range `1 ~ 10`. A single polling cycle can process up to 2,000 events across all pages. |

### SQL Safety Restrictions

The system strictly validates query statements. The following content is not allowed:

- Multi-statement queries containing `;`
- DML or DDL operations such as `INSERT`, `UPDATE`, `DELETE`, `DROP`, `ALTER`, `TRUNCATE`, and `CREATE`
- Lock operations such as `FOR UPDATE` and `LOCK IN SHARE MODE`
- Row limit keywords such as `LIMIT`, `OFFSET`, `FETCH`, and `TOP`; pagination is controlled automatically by the system
- Parameter placeholders such as `?`

:::tips
You only need to write the core query logic. Do not add `LIMIT` or `ORDER BY` to the SQL. The system automatically wraps your query as a subquery and adds pagination conditions and sorting based on the cursor columns.
:::

</div>

## III. Cursor Configuration <span id="cursor-configuration"></span>
---

<div class="md-block">

DB Pull uses keyset pagination to make sure each poll retrieves only incremental data. The system uses a combination of the time column and ID column to locate the last read position, then continues from that position on the next poll.

| Parameter | Required | Default | Description |
| :-: | :-: | :-: | :--- |
| **Time column** | Yes | - | Name of the time column used for sorting and pagination. It must appear in the SQL SELECT list. For example, `created_at` or `trigger_time`. |
| **ID column** | Yes | - | Name of the unique ID column used for deduplication and sorting within the same timestamp. It must appear in the SQL SELECT list. For example, `id` or `alert_id`. |
| **Initial time** | No | - | Starting time for the first poll. If left empty, polling starts from the earliest data. Supported formats include `2006-01-02 15:04:05` and `2006-01-02T15:04:05`. |

The automatically generated pagination query logic is as follows:

```sql
SELECT * FROM (your query) AS _src
WHERE (_src.time_column > ? OR (_src.time_column = ? AND _src.ID_column > ?))
ORDER BY _src.time_column ASC, _src.ID_column ASC
LIMIT ?
```

:::tips
Cursor positions are automatically persisted with a TTL of 30 days. If you change key configuration such as the database connection, SQL query, or cursor columns, the system detects the change and resets the cursor position automatically.
:::

</div>

## IV. Field Mapping <span id="field-mapping"></span>
---

<div class="md-block">

Column names returned by database queries usually differ from the field names in standard alert events, so they need to be converted through field mapping.

### Standard Field Mapping

Map columns from query results to standard alert event fields:

| Target field | Description |
| :--- | :--- |
| `event_status` | Alert status: Critical / Warning / Info / Ok |
| `title_rule` | Alert title |
| `alert_key` | Alert identifier |
| `description` | Alert description |
| `event_time` | Event time. Supports Unix timestamps, ISO 8601, and common datetime formats. |

Each field supports three mapping methods:

| Mapping method | Description | Example |
| :--- | :--- | :--- |
| `column` | Directly reference a column from the query result | `severity` |
| `template` | Use Go template syntax to concatenate multiple columns | `{{.host}} - {{.check_name}}` |
| `const` | Use a fixed value | `Critical` |

### Label Mapping

Labels support four modes:

| Mode | Description |
| :--- | :--- |
| `include_columns` | Select specified columns from the query result as labels. Only explicitly listed columns are extracted. |
| `all_columns` | Automatically use all columns that are not occupied by field mapping and are not cursor columns as labels. |
| `json_column` | Specify a column whose value is a JSON string. The system automatically parses it into label key-value pairs. |
| `manual` | Manually define the mapping between label names and source columns. Label names can differ from column names. |

The following examples are based on the same query:

```sql
SELECT id, created_at, severity, host, check_name, message, env, service, tags_json
FROM alert_records
WHERE status = 'open'
```

In this example, field mapping maps `severity` to `event_status`, `check_name` to `title_rule`, `id` to `alert_key`, `message` to `description`, and `created_at` to `event_time`. The cursor columns are `created_at` and `id`.

### Mode 1: Specified Columns, `include_columns`

Select specified columns from the query result as labels. Only explicitly listed columns are extracted; other columns are ignored.

```json
{
  "mode": "include_columns",
  "columns": ["env", "service", "host"]
}
```

Generated labels: `env=production`, `service=api`, and `host=web-01`.

### Mode 2: All Columns, `all_columns`

Automatically use all columns that are not occupied by field mapping and are not cursor columns as labels. This is suitable when there are many columns and you want to preserve all of them as labels.

```json
{
  "mode": "all_columns"
}
```

Using the query above as an example, `severity`, `check_name`, `message`, `created_at`, and `id` are already occupied by field mapping or cursors. The remaining columns, `host`, `env`, `service`, and `tags_json`, automatically become labels.

### Mode 3: JSON Column, `json_column`

Specify a column from the query result whose value is a JSON string. The system automatically parses it into label key-value pairs. This is suitable when the alert table already has a JSON-formatted label field.

```json
{
  "mode": "json_column",
  "json_column": "tags_json"
}
```

If the value of the `tags_json` column is `{"region":"cn-north","cluster":"k8s-prod"}`, the generated labels are `region=cn-north` and `cluster=k8s-prod`.

### Mode 4: Manual Mapping, `manual`

Manually define the mapping between label names and source columns. Label names can differ from column names. This is suitable when you need to rename labels or select only some columns.

```json
{
  "mode": "manual",
  "mapping": {
    "environment": "env",
    "service": "service",
    "host": "host"
  }
}
```

Generated labels: `environment=production`, `service=api`, and `host=web-01`. The left side is the label name, and the right side is the column name from the query result.

### Complete Field Mapping Configuration Example

```json
{
  "fields": {
    "event_status": { "type": "column", "value": "severity" },
    "title_rule": { "type": "template", "value": "{{.host}} - {{.check_name}}" },
    "alert_key": { "type": "column", "value": "id" },
    "description": { "type": "column", "value": "message" },
    "event_time": { "type": "column", "value": "created_at" }
  },
  "labels": {
    "mode": "include_columns",
    "columns": ["env", "service", "host"]
  }
}
```

</div>

## V. Severity Mapping <span id="severity-mapping"></span>
---

<div class="md-block">

External systems often use different field names and values for alert severity. Flashduty uses severity mapping to translate external values into the standard `Critical / Warning / Info` levels.

- The system reads the mapped `event_status` field value and looks up the corresponding Flashduty severity in the mapping table.
- Default mapping: `P1 → Critical`, `P2 → Warning`, `P3 → Info`. You can remove the default entries or add any number of custom entries.
- If the value does not match any key in the mapping table, it falls back to `Warning`.

Assume the `severity` column in your database stores values such as `critical`, `warning`, and `info`, which differ in capitalization from Flashduty standard values. You need to map the `severity` column to `event_status` in `field_mapping`, then convert the values through `severity_mapping`:

```json
{
  "field_mapping": {
    "fields": {
      "title_rule": { "type": "column", "value": "title" },
      "description": { "type": "column", "value": "description" },
      "event_status": { "type": "column", "value": "severity" }
    },
    "labels": {
      "mode": "include_columns",
      "columns": ["source", "status"]
    }
  },
  "severity_mapping": {
    "critical": "Critical",
    "warning": "Warning",
    "info": "Info"
  }
}
```

In this configuration, `field_mapping` extracts the database `severity` column as the `event_status` field, and `severity_mapping` converts the original column value, such as `critical`, into the Flashduty standard value `Critical`.

</div>

## VI. Default Route
---

<div class="md-block">

When creating a "Shared Integration" under **Integration Center => Alert Events**, you must configure a default route. Otherwise, newly pulled events are dropped. After creation, you can add finer-grained rules under **Route**.

A "Dedicated Integration" created under **Integrations** in a specific channel is automatically bound to that channel and does not need a separate default route.

</div>

## VII. FAQ
---

<details>
  <summary>Which databases are supported?</summary>

  Currently, **MySQL**, **PostgreSQL**, and **ClickHouse** are supported. If you use a database compatible with the MySQL or PostgreSQL protocol, such as TiDB, CockroachDB, or Aurora, it usually works as well.

</details>

<details>
  <summary>Why can't I write LIMIT in the SQL?</summary>

  The system uses keyset pagination to automatically control page size and page traversal logic. Your query is wrapped as a subquery, and the system adds `WHERE`, `ORDER BY`, and `LIMIT` conditions in the outer query. If you manually add row limits in the inner SQL, the pagination logic conflicts.

</details>

<details>
  <summary>Polling works, but no new alerts are created. Why?</summary>

  Check the following:

  1. Make sure `event_status` and `title_rule`, the two required fields, are correctly configured in field mapping.
  2. Check whether severity mapping is correct and whether the original value matches the mapping table.
  3. If `alert_key` is used, check whether it duplicates a previous alert. Events with the exact same status and content are deduplicated and dropped.
  4. Check whether the event matches any drop, inhibit, or silence rule.

</details>

<details>
  <summary>After I change the SQL or connection information, does polling start from the beginning?</summary>

  Yes. The system detects changes to key configuration such as the database connection, SQL query, and cursor columns. When a change is detected, the cursor is reset automatically, and the next poll starts again from the initial time or the beginning of the data.

</details>

<details>
  <summary>How is the database connection password stored?</summary>

  The password is encrypted with RSA when saved and decrypted only when polling runs. The system never stores or transmits the password in plaintext.

</details>
