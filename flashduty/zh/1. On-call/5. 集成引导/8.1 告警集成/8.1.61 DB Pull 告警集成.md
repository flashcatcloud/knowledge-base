Flashduty 主动按周期连接外部数据库执行 SQL 查询，将结果转换为标准告警事件，适用于告警数据存储在数据库中、无法主动推送的场景。

:::tips
DB Pull 适合告警数据存储在关系型数据库中、监控系统无法推送 webhook，或需要直接从业务表中抽取异常数据作为告警的场景。如果您的系统已经支持 webhook 推送，优先使用对应的推送式集成，延迟更低、资源开销更小。
:::

<div class="hide">

## 在 Flashduty
---

您可通过以下 2 种方式创建 DB Pull 集成，任选其一即可。

### 使用专属集成

当您不需要将告警事件路由到不同的协作空间，优先选择此方式，更简单。

<details>
  <summary>展开</summary>

  1. 进入 Flashduty 控制台，选择 **协作空间**，进入某个空间的详情页面。
  2. 选择 **集成数据** tab，点击 **添加一个集成**，进入添加集成页面。
  3. 选择 **DB Pull** 集成。
  4. 按下文「配置项」说明填写表单，点击 **保存**。

</details>

### 使用共享集成

当您需要根据告警事件的 Payload 信息，将告警路由到不同的协作空间，优先选择此方式。

<details>
  <summary>展开</summary>

  1. 进入 Flashduty 控制台，选择 **集成中心 ⇒ 告警事件**，进入集成选择页面。
  2. 选择 **DB Pull** 集成，并填写 **集成名称**。
  3. 配置 **默认路由**，选择对应的协作空间（集成创建后可前往 **路由** 配置更多路由规则）。
  4. 按下文「配置项」说明填写表单，点击 **保存** 完成。

</details>
</div>

## 概述
---

<div class="md-block">

通过配置数据库连接和 SQL 查询，Flashduty 会按照您设定的周期主动连接数据库执行查询，将每一行结果通过字段映射转换为符合 [标准告警事件](/zh/on-call/integration/alert-integration/alert-sources/standard%20alert) 协议的事件并写入告警通道。整个过程不需要您的系统具备推送能力，只要告警数据存储在可访问的数据库中即可。

### 适用场景

- 告警数据存储在 MySQL、PostgreSQL 或 ClickHouse 中
- 监控系统不支持 webhook 推送，只能通过数据库查询获取告警数据
- 需要直接从业务表中抽取异常数据作为告警
- 自研系统暂时无法改造为推送模式，希望先通过拉取方式快速接入

### 配置步骤

1. 进入 Flashduty 控制台，选择 **集成中心 → 告警事件**，搜索并选择 **DB Pull** 集成。
2. 配置数据库连接信息（详见 [数据库连接](#数据库连接)）。
3. 配置用于拉取告警数据的 SQL 查询（详见 [SQL 查询](#sql-查询)）。
4. 配置游标列，确保每次只拉取增量数据（详见 [游标配置](#游标配置)）。
5. 配置字段映射和严重程度映射（详见 [字段映射](#字段映射)、[严重程度映射](#严重程度映射)）。
6. 点击 **保存** 启用集成。您也可以在集成详情页手动触发一次拉取。

</div>

## 一、数据库连接 <span id="数据库连接"></span>
---

<div class="md-block">

| 配置项 | 必填 | 默认值 | 说明 |
| :-: | :-: | :-: | :--- |
| **数据库类型** | 是 | - | 支持 `MySQL`、`PostgreSQL`、`ClickHouse` 三种。 |
| **主机地址** | 是 | - | 数据库服务器地址，支持 IP 或域名。 |
| **端口** | 是 | - | 数据库端口，取值范围 `1 ~ 65535`。 |
| **用户名** | 是 | - | 数据库登录用户名。 |
| **密码** | 否 | - | 数据库登录密码，系统会加密存储。 |
| **数据库名** | 是 | - | 要连接的数据库名称。 |
| **连接参数** | 否 | - | 额外连接参数，Key-Value 格式。例如 MySQL 可传 `charset=utf8mb4`，PostgreSQL 可传 `sslmode=require`。 |

:::warning
出于安全考虑，Flashduty（非私有化部署）不允许连接内网地址（如 `127.0.0.1`、`10.x.x.x`、`192.168.x.x` 等）。请确保您的数据库可从公网访问，或使用私有化部署版本。建议为 Flashduty 创建一个只读账号，并仅授予所需表的 SELECT 权限。
:::

</div>

## 二、SQL 查询 <span id="sql-查询"></span>
---

<div class="md-block">

| 配置项 | 必填 | 默认值 | 说明 |
| :-: | :-: | :-: | :--- |
| **查询语句** | 是 | - | 用于拉取告警数据的 SQL `SELECT` 语句。必须以 `SELECT` 或 `WITH` 开头。 |
| **超时时间（秒）** | 否 | `5` | 单次查询的超时时间，取值范围 `1 ~ 10`。 |
| **拉取周期（秒）** | 是 | `300` | Flashduty 触发下一次拉取的间隔，最小值 30 秒。 |
| **每页行数** | 否 | `500` | 每次分页查询返回的最大行数，取值范围 `1 ~ 1000`。 |
| **最大分页数** | 否 | `5` | 单次拉取最多执行多少次分页查询，取值范围 `1 ~ 10`。单次拉取全部分页累计最多处理 2000 条事件。 |

### SQL 安全限制

系统会对查询语句进行严格校验，以下内容禁止使用：

- 多语句查询（包含 `;`）
- DML / DDL 操作（`INSERT`、`UPDATE`、`DELETE`、`DROP`、`ALTER`、`TRUNCATE`、`CREATE` 等）
- 锁操作（`FOR UPDATE`、`LOCK IN SHARE MODE`）
- 行数限制关键词（`LIMIT`、`OFFSET`、`FETCH`、`TOP`），分页由系统自动控制
- 参数占位符（`?`）

:::tips
您只需编写核心查询逻辑，不要在 SQL 中添加 `LIMIT` 或 `ORDER BY`。系统会自动将您的查询包装为子查询，添加基于游标列的分页条件和排序。
:::

</div>

## 三、游标配置 <span id="游标配置"></span>
---

<div class="md-block">

DB Pull 使用键集分页（Keyset Pagination）确保每次拉取只获取增量数据。系统根据时间列和 ID 列的组合定位上次读取的位置，下次拉取从该位置继续。

| 配置项 | 必填 | 默认值 | 说明 |
| :-: | :-: | :-: | :--- |
| **时间列** | 是 | - | 用于排序和分页的时间列名称，必须出现在 SQL 的 SELECT 列表中。例如 `created_at`、`trigger_time`。 |
| **ID 列** | 是 | - | 用于同一时间内去重和排序的唯一标识列名称，必须出现在 SQL 的 SELECT 列表中。例如 `id`、`alert_id`。 |
| **初始时间** | 否 | - | 首次拉取的起始时间点。不填则从最早的数据开始。格式支持 `2006-01-02 15:04:05` 或 `2006-01-02T15:04:05`。 |

系统自动生成的分页查询逻辑如下：

```sql
SELECT * FROM (您的查询) AS _src
WHERE (_src.时间列 > ? OR (_src.时间列 = ? AND _src.ID列 > ?))
ORDER BY _src.时间列 ASC, _src.ID列 ASC
LIMIT ?
```

:::tips
游标位置会自动持久化，TTL 为 30 天。如果您修改了数据库连接、SQL 查询或游标列等关键配置，系统会自动检测到变更并重置游标位置。
:::

</div>

## 四、字段映射 <span id="字段映射"></span>
---

<div class="md-block">

数据库查询返回的列名通常与标准告警事件的字段名不一致，需要通过字段映射进行转换。

### 标准字段映射

将查询结果中的列映射到标准告警事件的字段：

| 目标字段 | 说明 |
| :--- | :--- |
| `event_status` | 告警状态（Critical / Warning / Info / Ok） |
| `title_rule` | 告警标题 |
| `alert_key` | 告警标识 |
| `description` | 告警描述 |
| `event_time` | 事件时间（支持 Unix 时间戳、ISO 8601、常见日期时间格式） |

每个字段支持三种映射方式：

| 映射方式 | 说明 | 示例 |
| :--- | :--- | :--- |
| `column` | 直接引用查询结果中的列 | `severity` |
| `template` | 使用 Go 模板语法拼接多列 | `{{.host}} - {{.check_name}}` |
| `const` | 使用固定值 | `Critical` |

### 标签映射

标签支持四种模式：

| 模式 | 说明 |
| :--- | :--- |
| `include_columns` | 从查询结果中选取指定的列作为标签，只有明确列出的列才会被提取。 |
| `all_columns` | 将查询结果中所有未被字段映射占用且非游标列的列自动作为标签。 |
| `json_column` | 指定一个值为 JSON 字符串的列，系统自动解析为标签键值对。 |
| `manual` | 手动定义标签名与源列的对应关系，标签名可以与列名不同。 |

以下示例均基于同一个查询：

```sql
SELECT id, created_at, severity, host, check_name, message, env, service, tags_json
FROM alert_records
WHERE status = 'open'
```

其中字段映射已将 `severity` 映射为 `event_status`、`check_name` 映射为 `title_rule`、`id` 映射为 `alert_key`、`message` 映射为 `description`、`created_at` 映射为 `event_time`，游标列为 `created_at` 和 `id`。

### 模式一：指定列 `include_columns`

从查询结果中选取指定的列作为标签，只有明确列出的列才会被提取，其他列忽略。

```json
{
  "mode": "include_columns",
  "columns": ["env", "service", "host"]
}
```

生成的标签：`env=production`、`service=api`、`host=web-01`。

### 模式二：全部列 `all_columns`

将查询结果中所有未被字段映射占用且非游标列的列自动作为标签。适合列数较多、希望全部保留为标签的场景。

```json
{
  "mode": "all_columns"
}
```

以上述查询为例，`severity`、`check_name`、`message`、`created_at`、`id` 已被字段映射或游标占用，剩余的 `host`、`env`、`service`、`tags_json` 会自动成为标签。

### 模式三：JSON 列 `json_column`

指定查询结果中某一列，该列的值为 JSON 字符串，系统会自动解析为标签键值对。适合告警表中已有一个 JSON 格式标签字段的场景。

```json
{
  "mode": "json_column",
  "json_column": "tags_json"
}
```

如果 `tags_json` 列的值为 `{"region":"cn-north","cluster":"k8s-prod"}`，则生成标签：`region=cn-north`、`cluster=k8s-prod`。

### 模式四：手动映射 `manual`

手动定义标签名称和源列的对应关系。标签名可以与列名不同，适合需要重命名或只选取部分列的场景。

```json
{
  "mode": "manual",
  "mapping": {
    "环境": "env",
    "服务": "service",
    "主机": "host"
  }
}
```

生成的标签：`环境=production`、`服务=api`、`主机=web-01`。左侧为标签名，右侧为查询结果中的列名。

### 完整字段映射配置示例

```json
{
  "fields": {
    "event_status": { "type": "column", "value": "severity" },
    "title_rule": { "type": "template", "value": "{{.host}} - {{.check_name}}" },
    "alert_key": { "type": "column", "value": "id" },
    "description": { "type": "column", "value": "message" },
    "event_time": { "type": "column", "value": "created_at" }
  },
  "labels": {
    "mode": "include_columns",
    "columns": ["env", "service", "host"]
  }
}
```

## 五、严重程度映射 <span id="严重程度映射"></span>
---

<div class="md-block">

外部系统的告警级别字段名称、取值往往不一致，Flashduty 通过严重程度映射将外部值翻译为标准的 `Critical / Warning / Info`。

- 系统读取映射后的 `event_status` 字段值，按映射表查找对应的 Flashduty 严重程度。
- 默认映射：`P1 → Critical`、`P2 → Warning`、`P3 → Info`。您可以删除默认项、新增任意条目。
- 如果值没有命中映射表中的任何 Key，将统一降级为 `Warning`。

假设数据库 `severity` 列存储的值为 `critical`、`warning`、`info`，与 Flashduty 标准值的大小写不一致。需要在 `field_mapping` 中将 `severity` 列映射到 `event_status`，再通过 `severity_mapping` 转换值：

```json
{
  "field_mapping": {
    "fields": {
      "title_rule": { "type": "column", "value": "title" },
      "description": { "type": "column", "value": "description" },
      "event_status": { "type": "column", "value": "severity" }
    },
    "labels": {
      "mode": "include_columns",
      "columns": ["source", "status"]
    }
  },
  "severity_mapping": {
    "critical": "Critical",
    "warning": "Warning",
    "info": "Info"
  }
}
```

在这个配置中，`field_mapping` 将数据库的 `severity` 列提取为 `event_status` 字段，`severity_mapping` 再将列中的原始值（如 `critical`）转换为 Flashduty 标准值（`Critical`）。

</div>

## 六、默认路由
---

<div class="md-block">

新增「共享集成」（即在 **集成中心 ⇒ 告警事件** 中创建）时，必须配置默认路由，否则新拉取到的事件将被丢弃。集成创建后可在 **路由** 中追加更细粒度的规则。

「专属集成」（在某个协作空间的 **集成数据** 中创建）会自动绑定到所在空间，不需要单独配置默认路由。

</div>

## 七、常见问题
---

<details>
  <summary>支持哪些数据库？</summary>

  目前支持 **MySQL**、**PostgreSQL** 和 **ClickHouse** 三种数据库。如果您使用兼容 MySQL 或 PostgreSQL 协议的数据库（如 TiDB、CockroachDB、Aurora 等），通常也可以正常工作。

</details>

<details>
  <summary>为什么不能在 SQL 中写 LIMIT？</summary>

  系统使用键集分页自动控制每页行数和翻页逻辑。您的查询会被包装为子查询，由系统在外层添加 `WHERE`、`ORDER BY` 和 `LIMIT` 条件。如果您在内层 SQL 中手动添加了行数限制，会导致分页逻辑冲突。

</details>

<details>
  <summary>拉取正常但没有产生新告警？</summary>

  请检查以下几点：

  1. 确认字段映射中已正确配置 `event_status` 和 `title_rule` 这两个必填字段。
  2. 检查严重程度映射是否正确，原始值能否命中映射表。
  3. 如果使用了 `alert_key`，检查是否与之前的告警重复，状态和内容完全相同的事件会被去重丢弃。
  4. 检查是否匹配到了排除、抑制或静默规则。

</details>

<details>
  <summary>修改了 SQL 或连接信息后，会从头开始拉取吗？</summary>

  是的。系统会检测数据库连接、SQL 查询、游标列等关键配置的变更。当检测到变更时，游标会自动重置，下次拉取将从初始时间（或数据的起点）重新开始。

</details>

<details>
  <summary>数据库连接密码是如何存储的？</summary>

  密码在保存时会通过 RSA 加密，仅在执行拉取时解密。系统不会以明文形式存储或传输密码。

</details>
